## Bugs fixed

### Scoreboard ping showed "0" except briefly when the value changed
`LobbySidebar.refreshAll()` was calling `applyEnabledState(cfg)` every refresh tick, which iterated all online players and re-invoked `sidebar.addViewer(player)`. Minestom's `addViewer` always re-sends `DisplayScoreboardPacket` + every team-creation packet — including the ping line with its baseline `"Ping: 0"` prefix. The per-viewer `updatePing()` immediately after had a skip-if-unchanged guard, so when the ping value was stable across ticks the override never re-asserted and the client kept seeing `0`.

**Fix in [LobbySidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/LobbySidebar.java)**: track `lastEnabledApplied`; only run the bulk add/remove on actual transitions (i.e., config reload). Steady state is a single field compare.

### New joiners saw the *previous* player's ping on the sidebar
`PerViewerSidebar.updateLineContent(id, c, viewer)` called `ScoreboardLine.refreshContent(c)` via reflection. `refreshContent` delegates to `SidebarTeam.refreshPrefix(c)` which **mutates** the team's canonical `prefix` field. `Sidebar.addViewer(player)` then reads that field when sending `TeamsPacket.CreateTeamAction` to a new joiner — so the latest per-viewer prefix leaked to everyone added afterwards.

**Fix in [PerViewerSidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/PerViewerSidebar.java)**: dropped the `refreshContent` call entirely; `updatePrefix(c)` is non-mutating (it just builds the packet). Also cached the reflective `Method` at class load instead of looking it up per call.

### Every player's TAB ping icon stayed frozen at the value captured at join
Minestom auto-broadcasts `UPDATE_LATENCY` only on keepalive responses (~20 s). Until then, the per-player ping bars next to names in the TAB list showed whatever value was captured in the initial `ADD_PLAYER` snapshot — often near-zero on first join.

**Fix in [LobbyTabListManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/tab/LobbyTabListManager.java)**: added a 2 s broadcaster that batches all *changed* latencies into a single `PlayerInfoUpdatePacket` with `EnumSet.of(UPDATE_LATENCY)`. Unchanged latencies are skipped so the broadcast is silent when nothing moved.

### `{ping}` in TAB header/footer was bucketed to 50 ms (so a player at 23 ms saw `"0"`)
The render path grouped players by `(bypass, pingBucket)` and used `pingBucket * 50` as the displayed value. This was a performance trade-off from before the per-viewer skip-if-unchanged was added.

**Fix in [LobbyTabListManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/tab/LobbyTabListManager.java)**: render everything except `{ping}` once per `bypass` bucket; per-player work is now a single `String.replace("{ping}", actual)`. Skip-if-unchanged keeps idle ticks free.

### Launch-pad double jump silently stopped working after parkour
`LobbyJoinInitializer.setupState()` calls `player.setGameMode(default)`. Minestom's `Player.setGameMode` resets `allowFlying` to `gameMode.allowFlying()` — `false` for ADVENTURE/SURVIVAL. `LaunchPadManager.onJoin` fires on `PlayerSpawnEvent` (before the `setInstance().thenRun()` callback that lands in `setupState`) and sets `allowFlying = true`, but the subsequent `setGameMode` in the thenRun clobbered it.

**Fix in [LobbyJoinInitializer.java](src/main/java/ua/vsevolod/lobby/feature/lobby/player/join/LobbyJoinInitializer.java)**: re-assert `setAllowFlying(true)` + `setFlying(false)` after `setGameMode` in `setupState()`.

## What was optimized

| Area | Change | Impact |
|---|---|---|
| **Parkour instance leak** ([ParkourService.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourService.java)) | Every `/parkour` run created an `InstanceContainer` that was never unregistered. `stop()` now schedules a polling cleanup via `SchedulerManager.submitTask` that waits until the instance has no players then calls `unregisterInstance`. | Eliminates a real memory + tick-time leak that compounded with each parkour cycle. |
| **Music suppressor task leak** ([LobbyMusicManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/audio/music/LobbyMusicManager.java)) | `containsKey + put` race let two concurrent join events both spawn a task; only the second was tracked. Replaced with `computeIfAbsent` so creation + map insertion are atomic. | Each compromised join had been leaking one per-tick task forever — fixed by construction. |
| **Parkour HUD spam** ([ParkourSession.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourSession.java)) | `sendHud()` rebuilt the Component graph + sent an actionbar packet on every `PlayerMoveEvent` (5–10/s). Added `lastHudScore` / `lastHudSecond` short-circuit. | ~10× drop in HUD packet rate during steady-running parkour. |
| **Parkour stream allocation** ([ParkourSession.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourSession.java)) | `getNextTarget()` was `activeBlocks.stream().skip(1).findFirst().orElse(null)` — Stream + pipeline + Optional per tick. Now a two-step iterator advance. | Zero allocation per tick on the parkour fast path. |
| **NPC lookups O(N) → O(1)** ([NpcManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/interaction/npc/NpcManager.java)) | Added `IdentityHashMap<Entity, String>` reverse index for entity→id and `Map<String, NpcDefinition>` for id→def. Walked the full live list on every NPC right-click / attack before. | Hot interaction path drops from linear to constant. |
| **Sidebar concurrency** ([LobbySidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/LobbySidebar.java)) | `lastGlobalText` / `lastPingText` were plain `HashMap` accessed from scheduler + event threads. Switched to `ConcurrentHashMap`. | Removes a CME / table-corruption hazard under reload + join contention. |
| **Welcome hologram iteration** ([LobbyWelcomeHologramService.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/hologram/LobbyWelcomeHologramService.java)) | `refreshAll` walked `getOnlinePlayers()` and `continue`d for players without holograms. Now iterates `activeHolograms.entrySet()` directly. | At 500 online + 5 hologram viewers, drops 495 wasted iterations per call. |
| **QR pre-warm** ([LobbyQrMapService.java](src/main/java/ua/vsevolod/lobby/feature/lobby/interaction/qr/LobbyQrMapService.java), [LobbyModule.java](src/main/java/ua/vsevolod/lobby/bootstrap/module/LobbyModule.java)) | The `PACKET` static was initialized lazily — ZXing + Graphics2D ran inside the first join chain. Added `preinit()` and called it from `LobbyModule.load()`. | Lifts ~100 ms of Graphics2D work out of the critical join path. |

## Issues found but **not** fixed in this pass

Listed by severity. Full context in [AUDIT-2026-05-16.md](AUDIT-2026-05-16.md).

### Critical
- **CRIT-06** — `MongoParkourLeaderboardStore.submitResult` does `loadEntries()` after every write (full collection scan per submission).
- **CRIT-07** — `MongoParkourLeaderboardStore.updateEntries` is non-transactional `deleteMany + insertMany`; a crash between them wipes the leaderboard.

### High
- **HIGH-07** — `ParkourListener` PlayerMoveEvent fires globally for every move; should be scoped to the parkour instance via `EventNode` filter.
- **HIGH-08** — `NpcSkinResolver.resolveSync` blocks bootstrap on synchronous Mojang HTTP. **Spark profile confirmed this is 57 % of bootstrap time** — single biggest cold-start risk.
- **HIGH-09** — `LobbyTabListManager.scheduleRefresh` reads the interval once at startup; config-reload doesn't reschedule.
- **HIGH-10** — `LobbyJoinInitializer.enterLobby` is heavy and synchronous; NPC viewer-add happens inside a `synchronized` block so concurrent joins serialize.

### Medium / Low
- **MED-03..07** — `LobbyMusicManager` 7-map state could collapse into one record; `StatsBarService` uses `Player` as map key; `MinestomTagsWorkaround` listens to every outgoing packet; `LobbyProtocolWarningService` schedules a task per warned player; small unconditional work in `StatsBarService.tick`.
- **LOW-01..04** — config snapshot hoisting consistency; `Text.CACHE` wipe-on-overflow loses warm entries; static-field consistency; minor allocation observations.

### Newly found via Spark profile
- **ANVIL-01** — `ParkourService` creates instances without an explicit `ChunkLoader`, so Minestom defaults to `AnvilLoader`. Every chunk request probes the filesystem for non-existent `.mca` files. Visible at 0.30 s of FJ-pool CPU loadMCA / 0.27 s loadChunk / 0.18 s readChunkData during the profile window. Fix: pass a no-op `IChunkLoader` to `createInstanceContainer`. Concrete snippet in [SPARK-ANALYSIS-2026-05-16.md](SPARK-ANALYSIS-2026-05-16.md#2-anvilloader-is-running-on-parkour-instances--new).
- **JVM-01** — `-XX:MaxGCPauseMillis=130` is generous when a tick is 50 ms; a worst-case pause can eat 2–3 ticks. Recommend dropping to 50 (with staging validation).

## Important implementation notes

- **Toolchain bumped 25 → 26 in [build.gradle.kts](build.gradle.kts)** so the project builds in environments that have JDK 26 but not 25. Bytecode still targets Java 25 via `options.release.set(25)` — the produced JAR is unchanged.
- **Build verified**: `./gradlew shadowJar` ran cleanly after every change; the only compile note is a pre-existing deprecation warning in `TextHologramEntry`.
- **Profiling was idle**: the Spark profile we reviewed shows a near-idle server (TPS 20.00 throughout, 98.7 % tick-thread sleep, 0 full GCs in 268 s). Numbers from it for project-code self-time are tiny because the workload barely existed; the value is in the bootstrap path and the AnvilLoader finding. **Take a second profile under representative load** (≥50 players + an active parkour run) and compare.
- **The two reflection-based call sites in `PerViewerSidebar`** are intentional — we need to send a team-prefix-update packet to a single viewer without going through the broadcast path. Minestom does not expose this in the public API; the reflective call is wrapped in defensive try/catch and the cached `Method` lookup is now done once at class load. If Minestom adds a per-viewer API in a future release, that file can be deleted entirely.
- **`PlayerInfoUpdatePacket.Entry`** in this Minestom version requires *all* fields populated even when only one is read for the action. The TAB latency broadcaster fills in benign defaults for the unused fields (username, gameMode, listed=true, displayHat=true); clients only deserialise the latency field for `UPDATE_LATENCY` action.
- **Parkour instance cleanup uses `SchedulerManager.submitTask(Supplier<TaskSchedule>)`** which polls until the instance is empty then returns `TaskSchedule.stop()`. We can't unregister synchronously in `stop()` because the player is still inside the instance at that point (the teleport-out happens AFTER `stop()` in the call chain). The poll cost is one check every 20 ticks; negligible.
