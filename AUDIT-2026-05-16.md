# Lobby Optimization Audit — 2026-05-16

Full sweep of the `feature/admin-tools` branch (126 java files, ~10.5k LOC). This document captures every finding from the audit — what was fixed in this pass, what is recommended for later, and a few notes on patterns to keep an eye on. It is intentionally a snapshot dated to the day of the audit; do not edit in place — file a new dated audit when re-running.

Findings are tagged:

- **[FIXED]** — implemented in this audit.
- **[REC]** — recommended; left as-is to keep the diff focused and reviewable.
- **[NOTE]** — observation only; not a defect.

Severity: **CRIT** (correctness or unbounded growth) · **HIGH** (measurable runtime cost) · **MED** (sometime-cost or fragile) · **LOW** (cleanup).

## Top findings (fixed this pass)

### CRIT-01 [FIXED] · `PerViewerSidebar` mutated shared scoreboard state on every per-viewer update
**File:** [PerViewerSidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/PerViewerSidebar.java)
`updateLineContent(id, c, viewer)` called `ScoreboardLine.refreshContent(c)` via reflection, which delegates to `SidebarTeam.refreshPrefix(c)` and **mutates** the team's canonical `prefix` field. `Sidebar.addViewer(player)` reads that field when sending `TeamsPacket.CreateTeamAction` to a new joiner, so every new player saw "last-processed player's ping" on the scoreboard.

**Fix:** dropped the `refreshContent` call entirely; `updatePrefix(c)` is non-mutating (just builds the packet). Also cached the reflective `Method` at class load instead of re-looking it up per call.

### CRIT-02 [FIXED] · Sidebar ping flipped between correct value and `0`
**File:** [LobbySidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/LobbySidebar.java)
`refreshAll()` called `applyEnabledState(cfg)` every refresh tick, which iterated all online players and called `sidebar.addViewer(player)`. Minestom's `addViewer` always re-sends `DisplayScoreboardPacket` + every team-creation packet — including the ping line with its **baseline** `"Ping: 0"` prefix. The per-viewer `updatePing()` immediately after had a skip-if-unchanged guard, so when the ping value was stable across ticks the override never re-asserted and the client kept seeing `0`. Only when the ping value actually moved (between keepalives) did the player briefly see the real number.

**Fix:** track `lastEnabledApplied` and only perform the bulk add/remove on actual transitions (config reload). Steady state is a single field compare.

### CRIT-03 [FIXED] · Parkour instance leak
**File:** [ParkourService.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourService.java)
Every `/parkour` run called `instanceManager.createInstanceContainer(...)` to host the player. `stop()` removed entities but **never** unregistered the instance. The `InstanceManager` kept a strong reference to every past run forever — chunks, generator, block storage. A handful of cycles already shows on the heap; over a day of uptime this is a real memory leak and a TPS hit (every registered instance ticks).

**Fix:** `stop()` now schedules a polling cleanup task (`SchedulerManager.submitTask`) that waits until the instance has no players (the teleport-out happens AFTER `stop()` in the call chain) and then calls `unregisterInstance`.

### CRIT-04 [FIXED] · Double jump broke after parkour
**File:** [LobbyJoinInitializer.java](src/main/java/ua/vsevolod/lobby/feature/lobby/player/join/LobbyJoinInitializer.java)
`setupState()` calls `player.setGameMode(default)`. Minestom's `Player.setGameMode` resets `allowFlying` to `gameMode.allowFlying()` — `false` for ADVENTURE/SURVIVAL. `LaunchPadManager.onJoin` fires on `PlayerSpawnEvent` (before the `setInstance().thenRun()` callback that lands in `setupState`) and sets `allowFlying = true`, but the subsequent `setGameMode` in the thenRun clobbers it. Players returning from parkour silently lost the launch-pad double-jump.

**Fix:** in `setupState()`, re-assert `setAllowFlying(true)` + `setFlying(false)` after `setGameMode`.

### CRIT-05 [FIXED] · `LobbyMusicManager.ensureAmbientSuppressor` leaked tasks under concurrent joins
**File:** [LobbyMusicManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/audio/music/LobbyMusicManager.java)
`containsKey(uuid) + put(uuid, task)` was a TOCTOU race. Two concurrent join events for the same UUID could both pass the `containsKey` check and both schedule a suppressor; only the second was stored in the map. The first ran forever on every tick with no way to cancel — a tick-thread leak that compounded each time it tripped.

**Fix:** replaced with `computeIfAbsent` so task creation and map insertion are atomic.

### HIGH-01 [FIXED] · TAB player-ping icon stayed frozen at join value
**File:** [LobbyTabListManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/tab/LobbyTabListManager.java)
Minestom auto-broadcasts `UPDATE_LATENCY` only on keepalive responses (~20 s). Until then, the per-player ping bars next to names in the TAB list showed whatever value was captured in the initial `ADD_PLAYER` snapshot — often near-zero on first join.

**Fix:** added a 2 s broadcaster that batches all *changed* latencies into a single `PlayerInfoUpdatePacket` with `EnumSet.of(UPDATE_LATENCY)`. Unchanged latencies are skipped so the broadcast is silent when nothing moved.

### HIGH-02 [FIXED] · `{ping}` in TAB header/footer was bucketed (50 ms granularity)
**File:** [LobbyTabListManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/tab/LobbyTabListManager.java)
The render path grouped players by `(bypass, pingBucket)` and used `pingBucket * 50` as the displayed value — a player with 23 ms ping saw `"0"`, a player with 51 ms saw `"50"`. This was a performance trade dating from before the per-viewer skip-if-unchanged was added.

**Fix:** template renders everything except `{ping}` once per `bypass` bucket; per-player work is a single `String.replace("{ping}", actual)`. Skip-if-unchanged keeps idle ticks free.

### HIGH-03 [FIXED] · `ParkourSession.tick()` sent an actionbar packet per move (5–10/s/player)
**File:** [ParkourSession.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourSession.java)
The HUD shows score + elapsed seconds — content only changes on a score advance or a new whole second. The old code rebuilt the Component graph and pushed the packet on every PlayerMoveEvent regardless.

**Fix:** added `lastHudScore` / `lastHudSecond`; `sendHud()` short-circuits when neither has changed. Cuts the HUD packet rate by ~10× during steady parkour and eliminates the matching Component allocations.

### HIGH-04 [FIXED] · `ParkourSession.getNextTarget()` allocated a Stream per tick
**File:** [ParkourSession.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourSession.java)
`activeBlocks.stream().skip(1).findFirst().orElse(null)` ran on every PlayerMoveEvent during a parkour run. Each call allocated a Stream + skip pipeline + Optional. Replaced with a two-step iterator advance — zero allocation.

### HIGH-05 [FIXED] · `NpcManager.findIdByEntity` / `findById` were O(N)
**File:** [NpcManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/interaction/npc/NpcManager.java)
Both methods walked the live-NPC map / `lastApplied.npcs()` list on every NPC interaction. Trivial for the current handful of NPCs; bites linearly as the count grows.

**Fix:** added an `IdentityHashMap<Entity, String>` reverse index for entity→id (O(1) by reference) and a `Map<String, NpcDefinition>` for id→def (O(1) with case-insensitive fallback for admin commands).

### HIGH-06 [FIXED] · `LobbySidebar` ping/text caches were non-concurrent `HashMap`s
**File:** [LobbySidebar.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/sidebar/LobbySidebar.java)
`lastGlobalText` and `lastPingText` were plain `HashMap`s but were written from both the refresh-scheduler task and event handlers (show/hide on join/leave). A reload + join under load could race into a `ConcurrentModificationException` or corrupt the internal table.

**Fix:** switched both to `ConcurrentHashMap`.

### MED-01 [FIXED] · `LobbyWelcomeHologramService.refreshAll` walked all online players
**File:** [LobbyWelcomeHologramService.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/hologram/LobbyWelcomeHologramService.java)
Iterated `getOnlinePlayers()` and `continue`d when `activeHolograms` had no entry for the UUID. With 500 online and 5 hologram viewers, 495 iterations did nothing.

**Fix:** iterate `activeHolograms.entrySet()` directly; resolve player by UUID only for known viewers.

### MED-02 [FIXED] · QR map texture was rendered on the first player join
**File:** [LobbyQrMapService.java](src/main/java/ua/vsevolod/lobby/feature/lobby/interaction/qr/LobbyQrMapService.java)
The `PACKET` static was initialized lazily — the first call to `give(player)` triggered ZXing + Graphics2D inside the `LobbyJoinInitializer` chain. Not catastrophic, but worth lifting out of the critical join path.

**Fix:** added `preinit()` and called it from `LobbyModule.load()` so the render happens at server startup.

## Recommended next (not yet applied)

### CRIT-06 [REC] · `MongoParkourLeaderboardStore.submitResult` re-reads the entire collection after every write
**File:** [MongoParkourLeaderboardStore.java](src/main/java/ua/vsevolod/lobby/feature/parkour/leaderboard/MongoParkourLeaderboardStore.java)
`submitResult` does `upsertBestResult()` then `loadEntries()` — a full collection scan, returned to the caller, just to update the in-memory snapshot. Multiply by per-run submissions across the lobby and you have sustained DB load that scales with the leaderboard size, not the change rate.

**Recommend:** have `submitResult` return only the upserted entry; let `ParkourLeaderboardService` merge it into its in-memory list. Periodic `startAutoRefresh` already pulls the full snapshot every `LEADERBOARD_SYNC_MILLIS` for drift-correction.

### CRIT-07 [REC] · `MongoParkourLeaderboardStore.updateEntries` does non-transactional `deleteMany + insertMany`
A crash or connection drop between the two calls wipes the collection. Use a Mongo transaction (`session.withTransaction { ... }`) or rewrite as an `updateMany` upsert.

### HIGH-07 [REC] · `ParkourListener` PlayerMoveEvent fires globally for every move
**File:** [ParkourListener.java](src/main/java/ua/vsevolod/lobby/feature/parkour/ParkourListener.java)
The listener does a `service.getSession(player) == null` early return for non-parkour players — cheap individually, but called for every lobby-side move (5–10/s per player). At 500 lobby players that's ~2.5k–5k wasted invocations/s.

**Recommend:** restrict the `EventNode` using `EventNode.event("parkour", EventFilter.INSTANCE, instance -> instance instanceof ParkourInstance)` or similar. Saves a per-event hash lookup at scale and stops touching player objects entirely.

### HIGH-08 [REC] · `NpcSkinResolver.PlayerSkin.fromUsername` blocks startup on Mojang HTTP
**File:** [NpcSkinResolver.java](src/main/java/ua/vsevolod/lobby/feature/lobby/interaction/npc/NpcSkinResolver.java)
`resolveSync(username)` does a synchronous HTTP call to the Mojang profile API per NPC during bootstrap. Several NPCs with username skins serialize multi-second startup waits if Mojang is slow.

**Recommend:** route plain-username specs through the same async path as `url:` specs — spawn the NPC immediately with no skin and `updateSkin` when the resolve lands. Or pre-warm the cache from a `storage/npc-skins.json` file populated by admins.

### HIGH-09 [REC] · `LobbyTabListManager.scheduleRefresh` reads the interval once at startup
**File:** [LobbyTabListManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/tab/LobbyTabListManager.java)
Changing `update-interval-ms` via `/reload` has no effect — the task still fires at the original cadence until restart. Same for `LobbySidebar.startAnimationTask` / `startRefreshTask`.

**Recommend:** cancel + reschedule the task in the config listener when the interval changes, or have the scheduled task self-throttle by reading the current config and skipping ticks when elapsed-since-last < configured-interval.

### HIGH-10 [REC] · `LobbyJoinInitializer.enterLobby` is heavy and synchronous
Single-pass on the tick thread: show legacy holo, set display name + permission level, set gamemode, give items (config lookup, item building), show sidebar, render welcome hologram, render leaderboard hologram, show all NPCs. With many NPCs the `npcService.showTo(player)` step iterates every NPC and adds the player as viewer — that's already linear, and `showTo` is `synchronized` on `NpcManager`. Concurrent joins serialize.

**Recommend:** snapshot the live NPC list under lock, then call `addViewer` outside the synchronized block.

### MED-03 [REC] · `LobbyMusicManager` keeps six maps keyed by UUID
**File:** [LobbyMusicManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/audio/music/LobbyMusicManager.java)
`mutedPlayers`, `activeTasks`, `ambientSuppressors`, `playerQueues`, `lastPlayedTrack`, `currentTrack`, `playbackTokens` — seven independent per-player maps. Each disconnect handler must remember to clear all of them (currently does). A single-pass refactor into one `Map<UUID, PlayerMusicState>` would make leaks impossible by construction and halve the hash work.

### MED-04 [REC] · `StatsBarService` uses `Player` as a map key
**File:** [StatsBarService.java](src/main/java/ua/vsevolod/lobby/feature/admin/StatsBarService.java)
`Map<Player, BossBar>` holds strong references to the `Player` object. Cleanup is wired to `PlayerDisconnectEvent`, but if any other code path forgets to call into the service while a player vanishes (kick during configuration, etc.) the entry sticks and pins the Player.

**Recommend:** key by `UUID` like the rest of the codebase. Resolve the live `Player` from `ConnectionManager` when needed.

### MED-05 [REC] · `MinestomTagsWorkaround` listens to `PlayerPacketOutEvent`
**File:** [MinestomTagsWorkaround.java](src/main/java/ua/vsevolod/lobby/feature/lobby/player/workaround/MinestomTagsWorkaround.java)
`PlayerPacketOutEvent` fires for **every outgoing packet** for every player. The listener does a fast `instanceof TagsPacket` check, but this is the busiest event in the server. Even a cheap call adds up at scale.

**Recommend:** look into a Minestom-native packet-class filter (if the version exposes one) or wire the workaround at the connection level only for clients matching the protocol tag — register the listener lazily per-player in the configuration listener.

### MED-06 [REC] · `LobbyProtocolWarningService` schedules a 1 s repeating task per warned player
**File:** [LobbyProtocolWarningService.java](src/main/java/ua/vsevolod/lobby/feature/lobby/player/protocol/LobbyProtocolWarningService.java)
One task per warned player for 30 seconds. Negligible cost individually; if many old-version clients join in a wave, they all spin their own tasks. Consider a single per-server tick that walks a `Map<UUID, Integer secondsLeft>`.

### MED-07 [REC] · `StatsBarService.tick` recomputes the TPS label even when only RAM bars exist
The TPS section is computed unconditionally; the RAM section is guarded by `if (!ramBars.isEmpty())`. Symmetrical guards would shave a small amount of work in the RAM-only case.

### LOW-01 [REC] · `Sidebar` / `Tab` schedulers re-read config snapshot every call
Minor: `cfg = TabConfigSection.INSTANCE.current()` is just an `AtomicReference.get()` — cheap. Hoisting into a local is already done in the touched files. Flagging only as a pattern to keep consistent in new code.

### LOW-02 [REC] · `Text.CACHE` wipe-on-overflow loses warm entries
**File:** [Text.java](src/main/java/ua/vsevolod/lobby/util/Text.java)
At `CACHE_LIMIT` the whole map clears. The next 511 entries pay the parse cost again. With current usage this never trips, but if dynamic strings ever sneak into `Text.c(...)` you'll see a stutter at exactly 512 unique inputs and then again every 512 thereafter.

**Recommend:** swap to a small LRU. A 512-cap `LinkedHashMap` with `removeEldestEntry` behind a lock is enough.

### LOW-03 [REC] · `LobbyQrMapRenderer.createPacket` does 128×128 anti-aliased Graphics2D draws
QR rendering is fast but not trivial. Now precomputed at module init (MED-02), so currently fine — flag if `LobbyQrMapItem.URL` ever becomes per-player.

### LOW-04 [NOTE] · `ParkourService.PARKOUR_TEXT` is built with method chaining
Idle observation: the chained `.append(...)` allocates a few small components but only at class load.

## Allocation hotspots (per-tick or per-event)

After this pass:

- **TAB render:** two `StringBuilder`s for header/footer per bypass-bucket per tick, plus per-player `String.replace("{ping}", ...)` on the cached template. Skip-if-unchanged short-circuits the per-player path when ping is stable.
- **Sidebar refresh:** zero allocations on the unchanged path; one Component per changed global-line per tick; one packet per per-viewer ping change.
- **Parkour HUD:** zero allocations on unchanged ticks; one Component graph + one actionbar packet per score change OR new second.
- **Latency broadcast:** one `PlayerInfoUpdatePacket` per 2 s when at least one latency moved; nothing otherwise.
- **NPC lookups:** zero-alloc, O(1) reverse map.

Outstanding hotspots:

- `MongoParkourLeaderboardStore.loadEntries` allocates a fresh `ArrayList` + N entries per call. Acceptable for the 60 s refresh cycle; problematic when called inside `submitResult` (CRIT-06).
- `Component.text()...build()` chains in chat-message helpers — fine on user actions, would matter only if invoked from a tick loop (they are not).

## How to use this document

1. The **[FIXED]** entries are live in this branch — review the diffs and ship.
2. The **[REC]** entries are written so each one can be picked up independently. Pick by severity, not by section order.
3. Open new files? Re-run a targeted audit on:
   - Anything new with a tick listener or repeating task
   - Anything new keyed by `Player` (prefer `UUID`)
   - Anything new doing reflection or `printStackTrace()` in a hot path
