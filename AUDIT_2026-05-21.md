## 4. RestartCommand с отсчётом

Новый модуль `feature/admin/restart/`:

- **`RestartCountdownService`** — singleton с `AtomicReference<Running>`, отсчёт идёт
  на `BackgroundScheduler.SHARED`, тик-луп не задействован.
- **`RestartConfig`** — ConfigLib (см. п.8), backing-файл `config/restart.yml`.

Возможности:

- `/restart` — старт со значением `defaultCountdownSeconds` (по умолчанию 30).
- `/restart <сек>` — произвольная длительность (1…`maxCountdownSeconds`, по умолчанию 600).
- `/restart cancel` — отмена.

Эффекты по меткам (`ANNOUNCE_MARKS`): **300, 180, 60, 30, 15, 10, 5, 4, 3, 2, 1**.

| Метка | Title | Sound | Chat | Action bar |
|---|---|---|---|---|
| 300 / 180 / 60 | ✓ (длинный, ~2.5с) | `ENTITY_EXPERIENCE_ORB_PICKUP` | ✓ | ✓ |
| 30 | ✓ (длинный) | `ENTITY_EXPERIENCE_ORB_PICKUP` | ✓ | ✓ |
| 15 / 10 | ✓ (длинный) | `BLOCK_NOTE_BLOCK_BELL` | ✓ | ✓ |
| 5 / 4 / 3 / 2 | ✓ (короткий, ~0.8с) | `BLOCK_NOTE_BLOCK_BIT`, нарастающий pitch | — | ✓ |
| 1 | ✓ (короткий) | `ENTITY_ENDER_DRAGON_GROWL` (драматическая концовка) | — | ✓ |

Кик-экран — отдельный `Component` из `RestartConfig.kickMessage()`, поддерживает `&#RRGGBB`
и `\n` в YAML.

При `cancel()` всем игрокам отправляется `clearTitle()` и `Messages.warning("Перезапуск отменён.")`.

## 5. WriteBehindCache — per-key debounce

Это уже было реализовано корректно (`pendingTasks` — `ConcurrentHashMap<K, ScheduledFuture<?>>`),
но было плохо задокументировано. Сделано:

- Javadoc явно раскрывает per-key семантику — два игрока с разными UUID имеют независимые
  таймеры, действия одного не сдвигают сохранение другого.
- `scheduleFlush(K)` теперь использует `ConcurrentHashMap.compute()` — атомарный
  cancel-старого + schedule-нового, никаких orphan-future-ов.
- `evict(K)` тоже через `computeIfPresent()` — нельзя гонкой потерять отмену.

Для игровых данных (`PlayerPreferencesService`) задержка уже 60 секунд по UUID — соответствует
требованию ТЗ.

`util/WriteBehindCache.java`.

## 6. Fallback для старых протоколов без `text_display`

Создан пакет `feature/lobby/ui/hologram/render/` с чистой архитектурой:

```
DisplayType (enum)
   ├─ TEXT_DISPLAY
   └─ ARMOR_STAND

HologramRenderer (interface)
   ├─ TextDisplayRenderer   (1.19.4+, через TextDisplayMeta)
   └─ ArmorStandRenderer    (legacy, packet-based, NAMETAG_OFFSET = 1.7)

HologramRendererProvider
   ├─ forPlayer(Player)          → выбирает по ClientCompat.supportsTextDisplay
   ├─ displayTypeFor(Player)
   └─ rendererFor(DisplayType)
```

Renderer'ы — *stateless*; единственное состояние (позиция/текст/стиль) живёт в
`TextHologramEntry`, который теперь:

- Хранит `Map<UUID, DisplayType> viewerTypes` — какой backend выбрали при spawn для каждого зрителя.
- `show(p)` — выбирает renderer через provider, спавнит и кладёт в `viewerTypes`.
- `hide(p)` / `teleportAll` / `updateTextAll` / `updateStyleAll` — маршрутизируют пакеты через
  тот же тип, с которого зритель начинал (иначе на клиенте остался бы ghost-entity).
- Metadata-пакеты кешируются `EnumMap<DisplayType, …>` — один билд на тип за обновление.

`feature/lobby/ui/hologram/TextHologramEntry.java` похудел и стал чистым оркестратором —
никаких `if supportsTextDisplay` в боди.

## 7. Блок в void — мгновенно, под игроком, без смещения

`feature/lobby/world/protection/VoidProtectionListener.java`:

До:
```java
int x = (int) newPos.x();
int y = (int) newPos.y() + 1;   // выше игрока
int z = (int) newPos.z() - 1;   // сбоку
instance.setBlock(x, y, z, Block.GLASS);  // после телепорта
```

После:
```java
int x = (int) Math.floor(newPos.x());
int y = (int) Math.floor(newPos.y()) - 1;  // строго под ногами
int z = (int) Math.floor(newPos.z());       // без offset
instance.setBlock(x, y, z, Block.GLASS);   // ДО телепорта
```

Два независимых бага закрыты одновременно:

1. `(int)` cast на отрицательных координатах округляет к нулю → блок попадает на 1 клетку
   в сторону. `Math.floor()` округляет к минус-бесконечности → правильная клетка-колонка.
2. `setBlock` теперь до `teleport`, поэтому блок коммитится против текущей позиции игрока,
   а не той, куда его уже отправили.

## 8. Конфиги на ConfigLib

Добавлена зависимость `de.exlll:configlib-yaml:4.8.1` в `build.gradle.kts`.

Переведены на `@Configuration` + `@Comment` + `YamlConfigurations.update(Path, Class)`:

| Класс | Файл | Заметки |
|---|---|---|
| `config/ChatConfig.java` | `config/chat.yml` | flat-структура; статический `get()` / `load()` сохранён |
| `config/StorageConfig.java` | `config/storage.yml` | `playerPrefsMode` → `playerPrefs`, `leaderboardMode` → `leaderboard`; обновлены call-sites |
| `config/LoggingConfig.java` | `config/logging.yml` | `timezoneId` хранится строкой, парсится в `transient ZoneId` |
| `config/ProxyConfig.java` | `config/proxy.yml` | мигрировано с `.properties`; добавлены `velocityEnabled()/velocitySecret()/...` |
| `feature/admin/restart/RestartConfig.java` | `config/restart.yml` | новый — управляет `/restart` поведением |

Что даёт ConfigLib:

- `update(...)` сам создаёт файл с дефолтами, дописывает новые поля, сохраняет пользовательские
  значения и удаляет obsolete keys.
- `@Comment({...})` — комментарии прямо в коде, синхронизированы с полями.
- Сильная типизация — enum-ы (`StorageConfig.Mode`) автоматом, без `valueOf` и try/catch.
- При битом YAML — `catch + new ChatConfig()` (in-code defaults), сервер не падает.

Семь существующих `ConfigSection`-имплементаций (tab/sidebar/npcs/menus/holograms/join-items/
player-behavior) оставлены как есть — они уже имеют atomic-snapshot hot-reload и не страдают
теми проблемами, которые перечислены в ТЗ.
