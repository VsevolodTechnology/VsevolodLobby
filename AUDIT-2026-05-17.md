## Findings fixed in this pass

### HIGH-A · `ProxyOnlineService` accepted attacker-controlled server names

**File:** [ProxyOnlineService.java](src/main/java/ua/vsevolod/lobby/bootstrap/server/ProxyOnlineService.java)

`handleIncoming` (the `bungeecord:main` plugin-message handler) used to call
`counts.put(serverName, …)` for any string the client sent. A modded client can
write arbitrary bytes to `bungeecord:main` — so a single attacker could pump
thousands of unique `serverName` strings into the map and grow it without
bound. Heap leak + every subsequent caller walks a larger map.

**Fix:** drop the incoming message unless `serverName` is in the
`registeredServers` whitelist that the lobby explicitly registered at startup.
Untrusted clients can no longer add map entries.

### HIGH-B · `MenuManager.substitute` allocated a Collection iterator for a player count

**File:** [MenuManager.java](src/main/java/ua/vsevolod/lobby/feature/lobby/ui/menu/MenuManager.java)

The `{online}` placeholder substitution called
`MinecraftServer.getConnectionManager().getOnlinePlayers().size()` — which
returns an unmodifiable view backed by an iterator. Counting via `.size()`
allocates the view object even though Minestom maintains a cached `int`. Hit
per item, per lore line, per menu open.

**Fix:** swap to `connectionManager.getOnlinePlayerCount()` — a direct `int`
read, zero allocation.

### MED-A · `OpsStore.save()` was not atomic (privilege loss on crash)

**File:** [OpsStore.java](src/main/java/ua/vsevolod/lobby/feature/admin/OpsStore.java)

`Files.write(FILE, …, TRUNCATE_EXISTING)` truncates the destination *first*,
then streams names. A JVM kill / power loss between TRUNCATE and the last
flush leaves `ops.txt` with a partial list. Next server start reads the partial
file via `OpsStore.load()` → some ops silently lose `/op` rights.

**Fix:** write to `ops.txt.tmp` then `Files.move(tmp, file, REPLACE_EXISTING,
ATOMIC_MOVE)`. Readers see either the previous version or the new one — never
a torn half-write. Same pattern `ConfigManager` and `NpcConfigSection` already
use for their own writes.

### MED-B · `VersionGate.save()` was not atomic

**File:** [VersionGate.java](src/main/java/ua/vsevolod/lobby/feature/admin/VersionGate.java)

Same shape as MED-A — `Files.writeString` to a single target without a tmp
hop. On crash the gate file could come back malformed and either kick all
clients (if `min`/`max` got partial values) or silently disable the gate
(if `enabled=` line got truncated).

**Fix:** atomic tmp+move, same pattern.

## Findings looked at and confirmed NOT bugs

These were flagged by the broad-sweep audit run but turned out to be either
already correct or already covered by AUDIT-2026-05-16.

| Item | Status |
|---|---|
| `LobbyMusicManager.handleDisconnect` not cleaning all 7 maps | False alarm. Manual trace confirmed: `stopPlayback()` removes `activeTasks` + `currentTrack`, `cancelAmbientSuppressor()` removes `ambientSuppressors`, the remaining 4 are removed explicitly. All 7 maps are cleaned. The "consolidate into one PlayerMusicState" note from MED-03 (prior audit) still stands as a hygiene refactor — but it's not a leak today. |
| `LobbyTabListManager` calling `.size()` redundantly | False alarm. The collection is fetched once at the start of `updateAll()` and `.size()` is read off the cached reference — no duplicate work. |
| `ByteArrayWorldLoader` zip-slip | Correct as written: each entry is `.normalize()`'d and rejected via `startsWith(targetDir)` before extraction. |
| `NpcActionExecutor.runCommand` "command injection" | Not a vulnerability — `target` comes from NPC config which is owner-edited. No untrusted source feeds it today. |
| `HandshakeOverride` switch-fallthrough between TRANSFER → LOGIN | Intentional and correct — TRANSFER does its kick check then falls through to the shared `refreshServerInformation` below. |
| `OpCommand` shell-style escape via target | Minecraft chat protocol strips control characters before the command parser sees them. Owner-only command. No real attack vector. |

## Recommended (not yet applied, for next pass)

These remain open but are best held over to a targeted PR rather than
bundling with the audit-fix commit:

- **REC-A** (was MED-03) — `LobbyMusicManager` 7-map refactor to a single
  `PlayerMusicState` record. Prevents future cleanup-omission bugs by
  construction; not urgent because every map is currently cleaned correctly.
- **REC-B** (was MED-05) — `MinestomTagsWorkaround` listens to every outgoing
  packet for a tag-version check; consider a per-connection lazy listener
  scoped only to clients with the affected protocol tag.
- **REC-C** (was MED-03/4 in prior audit) — `LobbyMusicManager` still uses
  `Player` references inside `activeTasks` indirectly via the task lambda
  closure. UUID-only would be safer if a player ever vanishes via a
  non-`PlayerDisconnectEvent` path.
- **REC-D** — `NpcConfigSection.saveAndApply` and `MenusConfigSection.saveAndApply`
  do tmp+move already, but the *first-time template write* in `ConfigManager`
  uses tmp+move too — consistent good pattern. No fix needed; documenting for
  future reviewers.

## Coverage notes

Audited and clean:
- All static-init paths (`LobbyConfig.Settings.<clinit>`,
  `ParkourService.PARKOUR_TEXT`, hologram services) — no resource-grabbing
  before bootstrap completes.
- All `try { … } catch (Exception ignored)` blocks — verified each ignores a
  truly transient/expected exception, not silencing real bugs.

Skipped intentionally (out of scope or already audited):

- Network thread paths (would need a load-time Spark profile to reach
  conclusions — covered by SPARK-ANALYSIS-2026-05-16.md follow-up).
